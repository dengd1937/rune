# Design Review

对 writing-plans skill 输出的实施方案进行对抗性审查。**审查对象是方案文档，不是代码。**

在用户批准方案、开始编码之前，以怀疑者视角找出方案中不应被批准的理由。

## When to Activate

- writing-plans skill 输出方案后，用户审批前
- 方案涉及架构变更、新技术引入、多模块改动
- 用户主动调用

不强制执行，但建议在复杂方案上使用。

---

## 反逃避机制

### 审查跳过的合理化与反驳

| 借口 | 现实 |
|------|------|
| "方案已经很清晰了，不需要审查" | 方案越清晰越容易发现盲点。审查不是验证清晰度，是发现遗漏。 |
| "时间紧，先实现再说" | 未审查的方案问题在实现中暴露的代价远高于审查本身。 |
| "这个改动很小，不需要设计审查" | 小改动也可能有架构影响。审查成本低，忽略成本高。 |
| "我已经仔细想过了" | 自己审查自己的方案 = 盲点确认。独立视角才有价值。 |
| "审查会拖慢进度" | 修复设计缺陷比修复代码缺陷贵 10 倍。审查是加速器，不是减速器。 |

### Red Flags

- 跳过审查直接进入实现
- 审查只走了过场（没有发现任何问题）
- 审查发现的问题被标记为"后续优化"而非当场解决
- Review Packet 信息不完整就开始审查
- 没有引用方案具体内容就给出结论

→ 出现任何 Red Flag → 停止，重新执行完整审查流程。

## 审查立场

你不是方案的验证者，你是方案的挑战者。

```
默认怀疑：假设方案会以微妙、高成本、用户可感知的方式失败，直到证据表明相反。
不为意图加分：不认可"后续可以优化"、"先这样再说"、"理论上可行"。
必须有据：每个发现必须引用方案中的具体决策点。无法引用 = 不成立。
置信度校准：置信度 < 5 的发现降级到附录，不进入主报告。
```

以攻击者、混沌工程师、边界条件探测器的视角审视方案。

---

## Review Packet

审查开始前，必须构建标准化的 Review Packet。信息隔离是独立性的关键 — 审查者只看以下内容，不看 writing-plans 的推理过程。

```
## Review Packet

### 原始需求
[用户最初要求什么 — 原文，不做解读]

### 方案摘要
[writing-plans skill 输出的方案，包括架构设计、技术选型、实施步骤]

### 技术栈
[项目使用的语言/框架/数据库/核心依赖]

### 现有架构上下文
[系统现状 — 模块结构、数据流、已有约束]

### 影响范围
[涉及哪些模块/文件/接口/数据表]
```

如果调用方未提供完整 Packet，审查者应主动从项目中收集缺失信息（只读），然后再开始审查。

## 审查维度

逐维度审查，每个维度独立评分 0-10。

### 维度 1 — 架构合理性

```
关注点：
- 职责划分是否清晰，模块边界是否明确
- 依赖方向是否合理，有没有循环依赖风险
- 扩展性：未来需求变化时，方案是否需要大改
- 是否过度设计：方案复杂度是否超出需求复杂度
- 可逆性：决策是否容易回退，还是一旦实施就锁死

思考方式：
- 如果半年后需求变了，这个架构要改几个地方？
- 这个设计是"无聊但可靠"还是"花哨但脆弱"？
```

### 维度 2 — 实现可行性

```
关注点：
- 技术选型是否经过验证，有没有用不熟悉的技术解决已有成熟方案的问题
- 依赖风险：第三方库是否活跃维护，有没有已知的坑
- 边界条件：方案是否只覆盖了 happy path
- 数据一致性：并发、事务、幂等性是否被考虑
- 集成点：与现有系统的对接是否有兼容性风险

思考方式：
- 方案中哪些步骤最可能卡住？
- 有没有"看起来简单但实际很难"的部分？
```

### 维度 3 — 测试策略

```
关注点：
- 方案是否提到了测试思路，还是默认"后面再补"
- 关键路径是否可测：核心逻辑能否被单元测试覆盖
- 集成测试：模块间交互是否有测试方案
- 边界条件测试：异常输入、空值、并发场景是否被考虑
- 测试数据：是否需要特殊的测试数据或环境

思考方式：
- 如果这个方案实现了，TDD 的 RED 阶段应该写什么测试？
- 哪些路径最容易被遗漏？
```

### 维度 4 — 性能风险

```
关注点：
- N+1 查询、未分页查询、缺少索引
- 内存：大数据集是否会一次性加载到内存
- 缓存：是否有缓存策略，缓存失效怎么处理
- 并发：高并发下是否有瓶颈或竞态条件
- 外部调用：第三方 API 超时、重试、熔断是否被考虑

思考方式：
- 数据量扩大 100 倍，方案还能工作吗？
- 最慢的那条路径在哪里？
```

### 维度 5 — Scope Challenge

```
关注点：
- 方案是否超出了原始需求的范围
- 有没有更简单的方案能满足同样的需求
- 是否存在不必要的抽象或提前优化
- 分阶段实施是否可行：能否先交付核心价值，再迭代

思考方式：
- 删掉方案中的哪些部分，需求仍然能被满足？
- 有没有"一半工作量、80% 效果"的替代方案？
```

## 失败模式分析

审查完 5 个维度后，构建失败模式清单。针对方案中的每个关键决策，描述最可能的失败场景：

```
| 决策点 | 失败模式 | 触发条件 | 影响 | 缓解建议 |
|--------|----------|----------|------|----------|
| [方案中的具体决策] | [会怎么失败] | [什么情况下触发] | [影响范围和严重程度] | [建议] |
```

只列真实可能的失败模式，不编造不切实际的场景。

## 输出格式

所有内容在对话中输出，不写入文件。

```
## Design Review Report

### Review Packet 摘要
[一句话概括审查的方案]

### 各维度评分

| 维度 | 评分 | 关键发现 |
|------|------|----------|
| 架构合理性 | X/10 | [一句话] |
| 实现可行性 | X/10 | [一句话] |
| 测试策略 | X/10 | [一句话] |
| 性能风险 | X/10 | [一句话] |
| Scope Challenge | X/10 | [一句话] |

### 详细发现

[每个发现包括：]
- 维度：[属于哪个维度]
- 严重级别：CRITICAL / HIGH / MEDIUM / LOW
- 置信度：[1-10]
- 方案引用：[引用方案中的具体内容]
- 问题描述：[具体问题]
- 建议：[具体改进方向]

### 失败模式清单
[表格]

### 替代方案建议（如有）
[更简单或更合理的实现思路]

### 结论
判定：needs-attention / approve
理由：[一句话]

### 附录：低置信度发现（< 5）
[降级到此处的发现]
```

## Pass Criteria

审查完成的判断标准：

- [ ] Review Packet 已构建完整
- [ ] 5 个维度全部评分
- [ ] 每个发现都引用了方案中的具体内容
- [ ] 失败模式清单已输出
- [ ] 结论已给出（needs-attention / approve）
- [ ] 低置信度发现已降级到附录
