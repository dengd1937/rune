# Global Reviewer Prompt 模板

`code-review` skill 在 global 模式通过 `Task(subagent_type="general-purpose", model="opus")` 调度全局审查时使用此模板。
调度前替换：
- `{{PLAN_TEXT}}` → 完整计划文本（所有任务）
- `{{TASK_SUMMARIES}}` → 每个任务的 implementer 状态报告摘要（DONE / DONE_WITH_CONCERNS + 关键决策）
- `{{DIFF}}` → 使用 `git diff <first_base_SHA>..<last_head_SHA>` 获取全量 diff
- `{{BASE_SHA}}` / `{{HEAD_SHA}}` → 第一个任务的 base_SHA 和最后一个任务的 head_SHA

## 审查流程（4 步）

1. **收集上下文** — 读取完整计划、任务摘要、全量 diff（已在上方提供）；必要时读取关键文件的完整内容
2. **理解全局结构** — N 个任务构成了什么整体？模块边界在哪？数据如何在任务间流动？
3. **逐维度审查** — 按下方检查清单，从全局视角扫描
4. **报告发现** — 用下方输出格式。**只报告 >80% 置信的问题**

## 置信度过滤

- **报告**：>80% 置信的全局性问题
- **跳过**：单任务级别的问题（已被逐任务审查覆盖）
- **跳过**：风格偏好、命名建议（除非跨任务不一致）
- **合并**：同类问题（"3 个模块都缺错误处理" 而非 3 条独立报告）
- **优先**：跨模块集成缺陷、架构漂移、全局安全策略缺口

## 检查清单

### 跨任务集成一致性（CRITICAL — 必须检查）

不同 subagent 独立实现的模块之间是否对齐：

- **接口契约一致** — 模块间调用的参数类型、返回结构、错误格式是否统一
- **数据模型一致** — 不同任务操作同一数据时，schema、字段命名、nullable 约定是否一致
- **错误处理策略统一** — 是用异常、Result 类型、还是 error code？跨任务是否统一
- **类型/命名约定一致** — 同一概念在不同模块中是否用相同的名称和类型
- **依赖方向正确** — 无循环依赖，依赖方向与架构分层一致

```typescript
// BAD: 两个 subagent 对同一概念用不同类型
// Task 1 (user service):
interface User { id: string; email: string }
// Task 2 (order service):
type CustomerRecord = { userId: number; emailAddress: string }

// GOOD: 统一类型
// 共享 types:
interface User { id: string; email: string }
// 两个模块都引用同一类型
```

### 重复代码与冗余（HIGH）

- **跨任务重复逻辑** — 多个 subagent 是否独立实现了相同或高度相似的函数/组件
- **应提取但未提取的共享模块** — 重复逻辑是否应抽取为共享 utility / hook / service
- **配置散落** — 相同配置常量是否在多处重复定义
- **测试工具重复** — 多个测试文件是否重复了相同的 test helper / fixture

### 架构漂移（HIGH）

- **实现方向偏离计划** — 各任务的实现是否保持了计划定义的架构方向
- **模式不一致** — 一个 subagent 用 Repository 模式，另一个直接查数据库；一个用不可变数据，另一个用 mutation
- **抽象层级不一** — 各模块的抽象深度是否在同一层级
- **意外的架构决策** — 是否有 subagent 引入了计划外的新依赖、新模式、或新分层

### 全局安全策略（CRITICAL — 触发条件：涉及认证/输入/数据/外部 API）

- **安全策略全局一致** — 输入验证、认证检查、权限控制是否在所有入口统一实施
- **日志与监控覆盖** — 关键路径是否都有日志；日志是否统一格式；是否泄露 secrets
- **环境变量使用一致** — 是否所有模块都用相同方式读取和验证环境变量（无静默默认值）
- **CORS / CSRF 策略** — 所有端点的安全策略是否统一

### 全局可维护性（MEDIUM）

- **文件大小** — 实现完成后是否产生了超大文件（>800 行）需要拆分
- **导出清晰度** — 公共 API 是否有清晰的边界；内部实现是否被意外暴露
- **依赖合理性** — 是否引入了计划外的新依赖；现有依赖是否都被使用
- **TODO / FIXME 积压** — 是否有多个任务留下未处理的 TODO

### 端到端流程完整性（HIGH）

- **用户旅程可贯通** — 从入口到出口，数据流是否完整无断点
- **状态管理一致** — 不同模块对同一状态的管理方式是否兼容
- **错误传播正确** — 底层错误是否正确传播到顶层并给出有意义的用户反馈

## AI 多 subagent 实现的全局审查关注点

本次审查的代码由多个独立 AI subagent 分别实现。多 subagent 场景有 4 个特有风险：

1. **集成盲区** — 每个 subagent 只看到自己的任务，接口对齐问题只有全局审查能发现
2. **风格碎片化** — 不同 subagent 可能有不同的代码风格偏好，导致同一代码库中出现不一致
3. **全局约束遗漏** — 单个 subagent 不负责全局关注点（安全策略、日志、监控），容易集体遗漏
4. **重复发明** — 多个 subagent 独立解决相同的子问题，产出了重复实现

## 给主代理的设计说明（非输出内容）

在 subagent-driven-development 流程内全局审查的 HIGH = BLOCK 是**有意收紧**：与每任务审查（Step 4）保持一致的二态语义。全局审查发现的问题通常比单任务问题影响更大，因为它们影响多个模块的集成。
